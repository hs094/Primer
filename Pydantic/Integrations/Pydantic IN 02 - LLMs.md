# IN 02 — LLMs

🔑 **Key insight:** Pydantic's "LLMs" integration page isn't an SDK — it's a pair of `llms.txt` files that expose the docs in an LLM-friendly format so coding assistants can answer Pydantic questions accurately.

## What's actually shipped

The official integration page advertises two endpoints, not a Python module:

- **llms.txt** — *"a file containing a brief description of the project, along with links to the different sections of the documentation."*
- **llms-full.txt** — same structure, but every linked section is inlined. The docs warn this file *"may be too large for some LLMs."*

URLs:

```
https://docs.pydantic.dev/latest/llms.txt
https://docs.pydantic.dev/latest/llms-full.txt
```

The page positions itself between [[Pydantic IN 01 - Logfire]] and the dev-tools entries (see [[Pydantic IN 03 - Dev Tools]]).

## Limitations called out by the docs

The page is explicit about what these files *don't* do today:

- They *cannot* be "natively leveraged by LLM frameworks or IDEs."
- The suggested workaround is implementing an MCP server that parses `llms.txt` and serves the relevant sections to the model.

So this integration is currently a **documentation distribution format**, not a runtime API.

## What this page is *not*

Notably absent from the official integration page (so don't fabricate it):

- ⚠️ No SDK examples for `instructor`, `openai`, or `anthropic` clients.
- ⚠️ No `model_json_schema()` recipes specifically for tool-use / structured-output prompting.
- ⚠️ No "validation as a guardrail" patterns are codified here.

Those patterns live in the broader Python ecosystem (instructor, OpenAI's structured-outputs API, etc.), not in this Pydantic integration page itself.

## How Pydantic *does* show up around LLMs in practice

The general primitives that LLM-tooling libraries lean on are documented elsewhere in the Pydantic docs and crosslink into your other notes:

- **Schema export.** Every model exposes `Model.model_json_schema()` — the same schema [[Pydantic IT 01 - Architecture|generated by `GenerateJsonSchema`]]. LLM tool-call APIs accept JSON Schema directly.
- **Structured output validation.** Once the model returns text/JSON, run it through [[Pydantic CO 14 - Type Adapter|`TypeAdapter(MyModel).validate_json(raw)`]] — the resulting `ValidationError` is your guardrail; see [[Pydantic ER 01 - Error Handling]].
- **Strictness.** [[Pydantic CO 13 - Strict Mode]] is the right knob when you don't want Pydantic to coerce a string `"true"` into `True` from a model response.
- **Discriminated unions.** When prompting for one-of-many tool calls, [[Pydantic CO 06 - Unions]] (with a `Literal` discriminator) gives clean parsing and good error messages.

## Recommended consumption pattern

For a coding agent that needs reliable Pydantic answers:

1. Point the agent at `https://docs.pydantic.dev/latest/llms.txt` so it can pick which section to dive into.
2. Fetch `llms-full.txt` only if the agent's context window can take it.
3. Wrap retrieval in an MCP server if you want the assistant to do this lazily on demand.

For a *runtime* application that wants Pydantic-validated LLM output:

1. Define your output as a [[Pydantic CO 01 - Models|`BaseModel`]].
2. Pass `Model.model_json_schema()` to the LLM provider as the response schema / tool definition.
3. Validate the raw response with `Model.model_validate_json(...)` and treat `ValidationError` as a "retry the model" signal.

## Pairings worth knowing

- **Strict mode for tool calls.** When you accept LLM output verbatim, [[Pydantic CO 13 - Strict Mode]] avoids accidental coercions that mask schema-violation bugs (e.g. the model returning `"true"` where you expect a real `bool`).
- **Discriminated unions for tool dispatch.** [[Pydantic CO 06 - Unions]] with a `Literal` discriminator lets you parse "which tool did the model pick?" with one `model_validate_json` call and clean error reporting.
- **`TypeAdapter` over ad-hoc parsing.** For non-`BaseModel` outputs (a list of dicts, a `TypedDict`), [[Pydantic CO 14 - Type Adapter|`TypeAdapter`]] gives the same validate / dump / json-schema triple without a wrapper model.
- **Observability.** Combine with [[Pydantic IN 01 - Logfire|Logfire]] (`logfire.instrument_pydantic()`) to see every LLM-output validation failure as a span — the cheapest way to catch a model that's drifting off-schema in production.

## Watch-outs

⚠️ The `llms.txt` distribution is *documentation*, not code. Don't import it; fetch it.

⚠️ Don't assume the LLM honours the schema — always re-validate. Validation is the guardrail; the schema is just the prompt.

⚠️ `model_json_schema()` emits Pydantic's full JSON Schema (including `$defs`, `anyOf`, etc.). Some LLM APIs require a restricted subset — consult the provider's structured-output docs.

⚠️ For deterministic retries, log the raw LLM response *and* the `ValidationError` payload (see [[Pydantic ER 01 - Error Handling]]) — silently re-prompting hides drift in the model's adherence to your schema.

💡 **Takeaway:** The Pydantic page itself is just `llms.txt` distribution — the real LLM contract is `model_json_schema()` out, `model_validate_json()` back in, and a `ValidationError` retry loop.
